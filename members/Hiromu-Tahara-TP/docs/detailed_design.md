### `readDistance()` — （HC-SR04から距離取得）

**basic_design.md 2-2 との対応：** （基本設計書の関数一覧の説明を転記）

**引数：** `引数名`（型）: なし

**戻り値：** int型（なしの場合は void）

```
【処理の流れ】
1. TrigをLOW → 安定化
2. Trigを10μsだけHIGH → 超音波発射
3. pulseIn() で反射時間を取得
4. 距離 = (時間 / 2) * 0.0343 で計算
5. 計算結果をintに丸めて返す

【異常値処理】
- pulseIn() が0（反射なし）
- 距離が0cm以下
- 距離が400cm超

→ いずれも測定失敗と判断し「前回のdistanceを返す」

【理由】
- HC-SR04はノイズが多く、異常値を採用するとLEDが誤点灯するため




```

## 3. 重要ロジックの詳細設計

### 3-1. チャタリング防止（デバウンス処理）

> ※ ボタンを使う場合は必ず設計してください。

```
【目的】
ボタン入力は機械的なチャタリングにより ON/OFF が高速で揺れるため、
そのまま読み取ると誤動作（複数回のモード切り替え）が発生する。
これを防ぐためにデバウンス処理を行う。

【処理の流れ】
1. ボタンの現在の読み取り値（reading）を取得する
2. 前回の読み取り値（lastButtonState）と異なる場合、
   - lastDebounceTime に現在時刻を記録する
3. 現在時刻 - lastDebounceTime が DEBOUNCE_DELAY（30ms）を超えたら、
   - ボタンの状態が安定したと判断する
4. 安定した状態が LOW（押された）であれば true を返す
5. 最後に lastButtonState を更新する

【ポイント】
- ボタンは INPUT_PULLUP を使用しているため、押すと LOW になる
- 30ms の遅延時間は一般的なチャタリング除去に十分な値

【エラー・異常ケース】
- ボタンが物理的に押しっぱなしになった場合でも、
  デバウンス処理により「押された瞬間」だけを検出するため、
  モードが高速で切り替わり続けることはない。


```

---

### 3-2. millis() を使ったタイマー管理

```
【考え方】
  「前回実行した時刻」を記録しておき、「今の時刻 − 前回時刻 ≥ 周期」なら実行する。

【処理の流れ】

1. 距離測定（100ms周期）
   - now = millis()
   - now - lastMillisMeasure >= 100ms なら距離を再測定
   - 測定中もLED点滅は止まらない

2. LED点滅（距離に応じて100〜500ms）
   - now = millis()
   - now - blinkMillis >= blinkInterval ならLEDのON/OFFを反転
   - blinkInterval は距離に応じて変化（例：5cm以下で100ms）

【メリット】
  - 距離測定とLED点滅が同時に動作する
  - センサー値が滑らかに更新される
  - delay() によるフリーズが発生しない

```

## 6. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 実装上の問題確認

> 「この詳細設計書に書いた関数と処理フローをもとに Arduino でコードを書きます。バグになりやすい箇所・処理の抜け・型の問題はありますか？」

**AIの回答（要約）：**
 - 距離測定・LED制御・点滅制御がすべて millis() ベースで設計されており、処理が止まらない構造になっている点は良い。
 - readDistance() の異常値処理（前回値を保持）は HC-SR04 の特性に合っており妥当。
 - 閾値ロジックが明確に定義されているため、色の切り替え条件に曖昧さがない。
 - updateLED() の「毎回全LEDをOFFにしてから必要なLEDだけONにする」方針は、LEDの競合を防ぐうえで正しい。
 - グローバル変数の型は Arduino UNO のメモリ制約に対して十分軽量で問題なし。
 - 点滅モードのフラグ（isBlinkMode など）が明示されていないため、実装時に変数名を明確にしておくとバグを防げる。
 - 関数間の責務分離ができており、単体テストがしやすい構造になっている。

**対応した内容：**
 - 点滅モード用のフラグ（例：bool isBlinkMode）を追加し、doColorChange() と updateLED() の連携を明確化する。
 - 閾値の境界（30cm / 11cm / 6cm / 5cm）をコメントとしてコード内にも残す予定。
 - readDistance() の異常値検出時にデバッグ用の Serial 出力を追加し、原因調査しやすくする。
---

### Q2: 単体テスト仕様の確認

> 「Section 5 の単体テスト仕様書で、各関数の動作が正しく検証できていますか？テストが不足している項目や、境界値テストが必要な箇所を教えてください。」

**AIの回答（要約）：**
 - 出力系テストは距離ごとの色変化を網羅しており妥当。
 - 境界値（30cm / 11cm / 6cm / 5cm）ちょうどのときの動作確認があると、より堅牢なテストになる。
 - doBlinkControl() のテストが不足しているため、blinkInterval が正しく変化するかを確認する項目を追加するとよい。
 - readDistance() の異常値処理（前回値保持）が正しく動くかのテストがある点は良い。
 - 並行動作テスト（点滅中に距離変化）が含まれており、非ブロッキング処理の検証として適切。

**対応した内容：**
 - 境界値テスト（30cm / 11cm / 6cm / 5cm）を追加予定。
 - doBlinkControl() の blinkInterval が距離に応じて 500ms → 100ms に変化するかを確認するテストを追加予定。
