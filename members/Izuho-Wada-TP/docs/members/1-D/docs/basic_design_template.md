### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（スイッチを押す）──→ [超音波センサーで計測する] ──（表示する)


```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| （例）待機中 | センサー値を100msごとに監視する | 緑LED：500msで点滅 |
|待機中|スイッチが押されるのを待つ|出力なし|
|スイッチを押す|スイッチが押されたことを検知する|出力なし(時間に余裕があればプロペラを回す)|
|超音波センサーで計測する|距離を計測する|出力なし(時間に余裕があればプロペラを回す)|
|表示する|測定した距離を表示する|出力なし|

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| （例）LED点滅 | 500ms | **millis** | ボタン監視と並行するため |
| （例）センサー読み取り | 100ms | **millis** | 常時監視が必要なため |
| スイッチ入力監視 | 常時（毎ループ） | **millis** | delayを使うと押下タイミングを逃すため |
| チャタリング処理 | 20〜50ms | **millis** | 他処理と並行しながら安定判定するため |
| 超音波センサー計測 | 100ms | **millis** | 計測中もスイッチ監視を止めないため |
| プロペラ回転制御 | 常時（距離に応じて更新） | **millis** | PWM出力はdelayで止めるべきでないため |
| キーパッド入力監視 | 50〜100ms | **millis** | 入力を取りこぼさないため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| チャタリング | ボタンを押したとき | 前回入力から50ms未満の入力を検出 | デバウンス処理でスキップ |
| センサー異常値 | センサーが誤読したとき | 値が仕様範囲外（例: 0 or 400cm超） | その値を無視・前回値を使用 |
| アナログノイズ | アナログ入力を読んだとき | 閾値付近で値が揺れる | 閾値に±10〜20のバッファを設ける |
| 超音波センサー未接続 | Echo が常に 0 のまま | pulseIn() がタイムアウト | 距離を更新せず前回値を使用 |
| モーター過負荷 | プロペラが回らない/停止 | 回転数が変化しない or 電流増加 | PWM を下げて安全停止 |
| キーパッド誤入力 | ノイズで複数キーが同時反応 | 同時押し検出 | 入力を無視して再読み取り |
| 距離の急変 | 物体が突然消える/反射不良 | 前回値から±50cm以上の急変 | 異常値として無視し前回値を保持 |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
ピン配置は D2（スイッチ）、D7/D8（超音波）、D9（PWM）、D3〜D6（キーパッド）で 競合なし。

使用予定の部品はすべて標準ライブラリまたは軽量ライブラリで、ライブラリ競合の可能性は低い。

状態遷移は「待機 → 計測 → 表示」で 抜けがなく、戻りルートも明確。

すべての周期処理が millis() ベースで設計されており、delay() によるブロッキング問題は発生しない。

チャタリング対策・センサー異常値対策も設計済みで、実装時の誤動作リスクが低い。

**対応した内容：**
ピン競合チェックを行い、D0/D1（禁止ピン）や SPI/I2C ピンを避けて配置した。

状態遷移図を見直し、「待機 → 計測 → 表示」の流れと戻り条件を明確化した。

delay() を一切使用せず、すべての周期処理を millis() に統一した。

チャタリング処理（lastButtonMillis）とセンサー異常値処理（0cm/400cm超の無視）を追加した。

モーター過負荷・センサー未接続などの異常系も Section 4 に追加し、設計の抜けを解消した。
---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
スイッチ入力には チャタリング対策（lastButtonMillis を使った20〜50msのデバウンス処理） が設計されており問題なし。

すべての周期処理が millis() ベースで設計されているため、delay() によるブロッキングは発生しない。

超音波センサーはデジタル計測だが、異常値（0cm・400cm超）を無視する処理があり、ノイズ対策として十分。

アナログ入力（キーパッドなど）を使う場合は、閾値にバッファを持たせる設計があり、アナログノイズ対策も考慮済み。

全体として、入力の安定性と非同期処理が確保されており、実装時の誤動作リスクは低い。
**対応した内容：**
buttonState と buttonFlag を分離し、lastButtonMillis を用いた デバウンス処理 を追加した。

タイミング設計（2-3）で delay() を使用せず、すべて millis() に統一した。

超音波センサーの異常値（0cm・400cm超）を無視し、前回値を保持する処理を Section 4 に追加した。

アナログノイズ対策として、閾値に±10〜20のバッファを設ける方針を異常系に追加した。

これらの対策により、入力の安定性とリアルタイム性を確保した。
